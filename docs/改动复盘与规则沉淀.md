## 2026-06-30 CloudReadOnly PostgreSQL 权限漂移修复闭环

- 改动范围：AICopilot CloudReadOnly readonly role 授权 SQL、授权 apply/check 脚本、手动 provisioning workflow、服务器 `deploy-release.sh` preflight、本机 `local-release.sh` 部署文件同步、`CloudReadOnlyGovernedSchema` 元数据、LLM Text-to-SQL 输入、prompt version、权限错误分类、语义分析错误展示、架构/后端测试、项目规则和部署文档。未修改 CloudPlatform 业务代码、Cloud 业务数据、Cloud 写接口或 EdgeClient。
- 改动原因：线上 AICopilot 已能直连 Cloud PostgreSQL，但 readonly role 缺少 `mfg_processes` 的 `SELECT` 权限，导致“模切设备最近 1 天日志”类请求被包装成数据源不可用。根因不是网络或 OIDC 权限，而是治理白名单、授权脚本、探针和部署前检查没有形成同一权威源，历史 workflow 内联 GRANT 清单容易漂移。
- 影响面：CloudReadOnly 仍只允许读取 `devices`、`mfg_processes`、`device_logs`、`hourly_capacity`、`pass_station_records` 五张白名单表，不新增任何 Cloud 写权限；SQL guard 和 BusinessQuery safety policy 仍使用表/列白名单；部署启用 direct DB 时会先验证 readonly role 权限，失败即停止部署。
- 线上止血：已在 `10.98.90.154` 的 Cloud PostgreSQL 容器中对 `aicopilot_cloud_readonly` 执行表级 `GRANT SELECT ON TABLE public.mfg_processes`，随后验证五张白名单表 `SELECT` 全部为 true，五张表 `INSERT/UPDATE/DELETE` 全部为 false，`public` schema `CREATE` 为 false。
- 关键实现：新增 `deploy/enterprise-ai/cloud-readonly/apply-readonly-grants.sql` 和 `check-readonly-grants.sql` 作为唯一授权/探针 SQL；新增 `deploy/enterprise-ai/scripts/apply-cloud-readonly-grants.sh` 和 `check-cloud-readonly-grants.sh`，支持 dry-run、解析服务器 `.env`、定位 Cloud PostgreSQL 容器并 fail-fast 校验；`aicopilot-provision-cloud-readonly-db-role` workflow 改为 checkout 后读取同一 SQL 文件，禁止继续维护内联 GRANT 清单；`deploy-release.sh` 在 direct DB 启用时先执行 readonly preflight；`local-release.sh` 发布前同步 `cloud-readonly/` 和脚本到服务器。

- 权限诊断：CloudReadOnly 执行异常如果是 `permission denied`，只暴露治理白名单内的表名，例如 `mfg_processes`；连接串、role、密码、SQL 原文和非白名单对象不会进入用户可见诊断。语义分析层对该类错误返回“CloudReadOnly 只读权限不足”而不是泛化为数据源不可用。
