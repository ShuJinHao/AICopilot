# AICopilot A助理产品化收口计划

Updated: 2026-05-14

## 1. 背景判断

当前动态模型、模型连通测试、真实执行模型展示已经进入可验收阶段，但产品层还有几类问题没有收口：

- 提示词身份不稳定。用户期望产品身份统一为“A助理”，不应再出现“朝小夕”“朝夕”等旧称呼，也不应让模型随意自称为 Claude、DeepSeek 或其他供应商身份。
- 历史上下文不可配置。路由阶段目前存在固定历史条数，最终回答阶段也缺少明确的历史轮次注入策略，管理员无法控制上下文成本与回答连续性。
- 会话历史可读性差。会话标题默认是“新会话”，列表缺少首问摘要、更新时间、删除/重命名等基础管理能力。
- MCP/Agent 能力边界还不清楚。当前系统有 MCP 工具框架、工具暴露模式、风险等级和审批链路，但还不是 Claude Code / Codex 那种本地自主执行 Agent。
- 聊天界面信息结构偏弱。用户需要看到“当前选择模型”“本条真实模型”“路由结果”“工具执行过程”“数据来源”“生成产物”，而不是只看到一串消息。

本计划只覆盖 `AICopilot`，不修改 `IIoT.CloudPlatform` 和 `IIoT.EdgeClient`。AICopilot 仍然保持对 Cloud 业务数据只读，不通过 MCP、Agent workflow 或隐藏适配器写入 Cloud 业务数据。

## 2. 目标

本轮目标不是继续散点加功能，而是把 AICopilot 收口成一个可验收、可诊断、可继续扩展的“A助理”产品形态。

需要达到的结果：

- 系统提示词、路由提示词、最终回答要求统一使用“A助理”身份。
- 管理员可以配置历史上下文轮数和 token 使用边界。
- 会话列表不再全是“新会话”，能够看到可区分的标题和更新时间。
- 普通用户能明确知道每条回答实际用了哪个回答模型、哪个路由模型、是否用了工具、是否生成了产物。
- MCP/Agent 能力以受控任务执行方式开放，不变成无限制 shell 或无限制文件系统写入口。

## 3. 非目标

以下内容不放入本轮：

- 不做 Cloud 业务写入能力。
- 不开放任意 shell 执行。
- 不让 AICopilot 绕过 Cloud 现有权限、审批、设备和配方规则。
- 不做按租户、按角色的模型授权细分。
- 不做完整 IDE 级代码代理。第一版只做受控文件产物和受控 MCP 任务。
- 不把提示词写成隐藏硬编码。提示词应有清晰来源，可由管理员查看或重置。

## 4. 设计原则

### 4.1 提示词是受治理配置

提示词不能只靠代码里的散落字符串。

需要建立一套“内置提示词基线”：

- `AAssistant.DefaultChat`：默认对话助手模板。
- `AAssistant.IntentRouting`：路由识别模板。
- `AAssistant.DataAnalysis`：数据分析回答约束。
- `AAssistant.McpTask`：受控工具任务执行约束。

内置模板需要包含版本号、用途、默认启用状态和可重置能力。管理员可以编辑，但系统必须能检测关键安全边界是否被删除。

### 4.2 历史上下文是运行配置

历史上下文不应写死在执行器里。第一版建议做全局管理员配置：

- 路由历史消息数，例如 `4`、`8`、`12`。
- 最终回答历史轮数，例如 `3`、`5`、`10`。
- 历史消息最大 token 预算。
- 历史列表默认加载数量。

后续如果需要，再扩展成“全局默认 + 模板覆盖”。第一版不建议直接做复杂覆盖体系。

### 4.3 Agent 能力必须受控

AICopilot 可以朝 Agent 化发展，但必须分层：

- 路由只负责判断意图。
- RAG 只负责知识检索。
- DataAnalysis 只负责只读结构化分析。
- MCP 只负责调用被管理员允许的工具。
- Human-in-the-loop 负责控制风险动作。

文件夹和文件生成属于本地副作用，默认需要审批和审计。第一版只允许写入管理员配置的工作区目录或系统产物目录，不允许模型自选任意路径。

### 4.4 UI 展示系统真相

不要让用户通过问模型“你是什么模型”来判断系统行为。前端必须展示系统记录的真实元数据：

- 顶部模型选择器表示“下一条消息将使用的回答模型”。
- 消息角标表示“这一条回复实际使用的回答模型和路由模型”。
- 工具时间线表示“这一条回复调用了哪些工具，是否审批，是否成功”。
- 产物区表示“这一条任务生成了哪些文件、图表、表格或报告”。

## 5. 分阶段方案

### Phase 1: 提示词与身份治理

改造内容：

- 新增内置提示词定义和版本管理。
- 将默认身份统一为“A助理”。
- 禁止出现“朝小夕”“朝夕”“朝小”“小夕”等旧身份。
- 最终回答提示词增加身份边界：不能自称为供应商模型，不能声称拥有未授予的系统权限。
- 路由提示词继续要求 JSON 输出，不承担最终回答身份。
- 管理员配置页增加“恢复内置模板”或“应用默认 A助理模板”入口。

需要注意：

- 代码仓库里当前未直接检出旧称呼，旧身份更可能来自数据库已有模板或模型幻觉。
- 因此必须处理数据库里的现有模板，单改源码不够。

验收标准：

- 新建模板和重置模板后，系统提示词统一称为“A助理”。
- 数据库内置模板不再包含旧身份。
- 用户问“你是谁”时，提示词层面要求回答为“A助理”。
- 单元测试扫描内置提示词，禁止旧称呼。

### Phase 2: 历史上下文与会话标题

改造内容：

- 新增运行配置，例如 `ChatRuntimeSettings`。
- 管理员配置路由历史消息数、最终回答历史轮数、历史 token 上限。
- 路由执行器不再硬编码历史条数。
- 最终回答构建器注入可控历史摘要或历史消息。
- 新会话首条消息后自动生成标题。
- 会话列表增加更新时间、最后一条消息摘要、消息数量。
- 支持重命名、删除会话，删除需要确认。

建议第一版标题策略：

- 首条用户消息前 24 到 32 个字符作为标题。
- 去掉换行和多余空格。
- 为空时仍显示“未命名会话”。
- 后续可以再加 AI 摘要标题，不作为第一版阻塞。

验收标准：

- 管理员修改历史轮数后，下一次请求生效。
- 最终回答能引用同一会话前文。
- 会话列表不再出现多条无法区分的“新会话”。
- 删除/重命名不会影响其他用户会话。

### Phase 3: 受控 Agent 文件产物能力

改造内容：

- 定义本地文件产物能力，不开放任意 shell。
- 管理员配置产物根目录或应用内产物存储。
- MCP 工具暴露时标记能力类型：只读查询、诊断、建议、本地产物写入、外部副作用。
- 本地产物写入默认 `RequiresApproval`。
- 写入前展示计划：目标目录、文件名、文件类型、预计内容摘要。
- 写入后落审计：用户、会话、工具、路径、文件 hash、时间、审批结果。
- 前端展示产物入口，支持打开或下载生成文件。

建议第一版能力：

- 创建任务产物文件夹。
- 生成 Markdown 报告。
- 生成 CSV/JSON 数据文件。
- 生成简单图表数据文件。

暂不做：

- 任意命令执行。
- 任意目录写入。
- 直接修改 Cloud 或 Edge 源代码。
- 自动提交 Git。

验收标准：

- 用户要求“处理数据并生成文件夹”时，系统先给出执行计划。
- 审批通过后，只能写入允许目录。
- 审批拒绝后，不产生文件。
- 审计记录能追踪每次写入。

### Phase 4: 聊天界面重构

改造内容：

- 左侧会话列表显示标题、更新时间、最后摘要、搜索入口。
- 中间消息区区分用户消息、助手回答、工具过程、产物卡片。
- 顶部模型选择器明确标注“下一条回答模型”。
- 每条助手消息显示真实回答模型、路由模型、意图标签。
- 右侧信息栏改为可折叠的“运行详情”，显示数据来源、工具调用、审批状态、产物。
- 空会话页改成“A助理工作台”的任务入口，不再只放制造业示例。

验收标准：

- 用户能从界面直接看出当前会话、当前模型、真实执行模型和工具执行状态。
- 历史会话可区分、可搜索、可管理。
- 移动端不挤压主聊天区。

### Phase 5: 质量门禁与回归

需要补齐测试：

- 提示词治理测试：内置模板无旧身份，包含 A助理身份，危险边界不能被删。
- 历史上下文测试：配置不同历史轮数时，请求上下文符合预期。
- 会话标题测试：首问后生成标题，重命名后不被覆盖。
- Agent 文件产物测试：未审批不写文件，越界路径被拒绝，审批通过写入允许目录。
- 前端测试：历史列表、模型角标、工具时间线、产物卡片。

基础验证命令：

- `dotnet build src/hosts/AICopilot.HttpApi/AICopilot.HttpApi.csproj`
- `dotnet test src/tests/AICopilot.BackendTests`
- `dotnet test src/tests/AICopilot.ArchitectureTests`
- `cd src/vues/AICopilot.Web && npm run test:unit`
- `cd src/vues/AICopilot.Web && npm run build`

Docker/Aspire 集成测试如果因为本机 Docker 不可用阻塞，需要在 PR 里明确标注环境原因。

## 6. 需要进一步确认的问题

### 6.1 数据库已有模板如何处理

候选策略：

- 只覆盖系统内置模板，保留管理员自建模板。
- 新增一套 A助理模板，让管理员手动切换。
- 全量替换现有模板。

建议选择“只覆盖系统内置模板，保留管理员自建模板”。这样既能修掉默认体验，又不会意外覆盖用户后续自建模板。

### 6.2 文件产物能力放在哪里

候选策略：

- 写入管理员配置的本地工作区目录。
- 写入应用管理的产物目录，通过前端下载。
- 暂不开放文件写入，只返回文本。

建议第一版使用“应用管理的产物目录”。如果需要真实文件夹路径，再加管理员配置的本地工作区目录。

### 6.3 历史上下文配置粒度

候选策略：

- 全局配置。
- 按模板配置。
- 全局默认 + 模板覆盖。

建议第一版先做全局配置，避免配置系统过早复杂化。等模板稳定后，再加模板覆盖。

### 6.4 Agent 是否允许自动连续多步执行

候选策略：

- 每个副作用步骤都审批。
- 同一任务审批一次，允许执行多个已声明步骤。
- 只允许只读多步，写入步骤单独审批。

建议第一版选择“只读多步可以自动执行，写入步骤单独审批”。

## 7. 推荐执行顺序

1. 先做 Phase 1 和 Phase 2。它们直接解决当前体验问题：身份不稳定、历史不可控、会话列表不可读。
2. 再做 Phase 3。Agent 文件产物涉及副作用、审批、审计，必须单独验收。
3. 最后做 Phase 4。UI 重构应基于已经稳定的后端元数据和产物协议。
4. Phase 5 贯穿每个阶段，阶段完成后更新交付记录。

## 8. 对外讨论重点

如果需要在 GitHub 或网页上继续评审，重点讨论这几个问题：

- A助理身份是“固定产品身份”，还是允许管理员自定义助手名称。
- 历史上下文第一版做全局配置是否足够。
- 文件产物应该先走应用产物目录，还是直接允许管理员指定本地工作区。
- Agent 自动执行的审批粒度应该多严格。
- 聊天 UI 是先小步增强，还是直接重做整体布局。
