# Phase 3.8 环境与基线

## 1. 适用范围

- 本文档只适用于 `AICopilot` 的 Phase 3.8 内部演示与验收。
- 当前仍使用 `cloud-device-semantic-sim` 作为演示业务只读源，不进入真实云端只读源替换。
- 本阶段目标是稳定演示、稳定验收、稳定回归，不新增前端 E2E 框架。

## 2. 演示基线

- 启动入口固定为 `AICopilot.AppHost`。
- 业务问答固定使用 `cloud-device-semantic-sim`。
- 审批示例固定使用诊断建议场景，不再使用任何控制型动作问句。
- 审批触发示例问句固定为 `please prepare a diagnostic checklist for device DEV-001`。
- 普通用户账号不预置，由管理员在治理页现场创建一个 `User` 角色用户。
- 手工演示允许使用外部可访问的 OpenAI-compatible 聊天模型接口。
- 自动回归仍使用测试专用 `FakeAiProviderHost`，不把它当作手工演示依赖。

## 3. 本地依赖

- Docker Desktop 已启动。
- .NET SDK 10 可用。
- Node.js 版本满足仓库要求。
- `npm` 可用。

## 4. 推荐启动方式

首次演示前先安装前端依赖，然后使用 `AppHost` 拉起本地栈。

```powershell
$env:BootstrapAdmin__UserName = "admin"
$env:BootstrapAdmin__Password = "Password123!"
$env:AppHost__PersistentContainers = "true"

cd "C:\Users\jinha\Desktop\产线系统架构升级\1\AICopilot\src\vues\AICopilot.Web"
npm install

cd "C:\Users\jinha\Desktop\产线系统架构升级\1\AICopilot"
dotnet run --project ".\src\hosts\AICopilot.AppHost"
```

说明：

- `BootstrapAdmin__UserName` 与 `BootstrapAdmin__Password` 是 Bootstrap 管理员来源。
- `AppHost__PersistentContainers=true` 用于固定演示期的数据基线。
- 如当前机器已有稳定管理员口径，可覆盖上面的推荐值；本文档默认按 `admin / Password123!` 编写。

## 5. 启动后应看到的资源

`AppHost` 正常启动后，应能看到以下资源处于可用状态：

- `postgres`
- `eventbus`
- `qdrant`
- `aicopilot-migration`
- `aicopilot-httpapi`
- `rag-worker`
- `aicopilot-webui`
- 数据库资源 `ai-copilot`
- 数据库资源 `cloud-device-semantic-sim`

如果 `aicopilot-webui` 没有正常拉起，可使用下面的回退方式。

```powershell
$env:AppHost__EnableWebUi = "false"

cd "C:\Users\jinha\Desktop\产线系统架构升级\1\AICopilot"
dotnet run --project ".\src\hosts\AICopilot.AppHost"

cd "C:\Users\jinha\Desktop\产线系统架构升级\1\AICopilot\src\vues\AICopilot.Web"
npm run dev
```

## 6. 账号基线

### 6.1 手工演示

- 管理员账号来源：`BootstrapAdmin__UserName`、`BootstrapAdmin__Password`
- 推荐演示值：`admin / Password123!`
- 普通用户账号来源：管理员在 `/access` 页面现场创建，角色固定为 `User`

### 6.2 自动回归

- 后端集成回归通过 `AICopilotAppFixture` 注入同名环境变量。
- 当前测试基线账号固定为 `admin / Password123!`

## 7. 配置基线

手工演示前需要准备 5 类正式配置：

- 1 个可用的语言模型
- 1 个通用会话模板
- 1 个 `IntentRoutingAgent`
- 1 条针对 `DiagnosticAdvisorPlugin / GenerateDiagnosticChecklist` 的审批策略
- 1 个指向 `cloud-device-semantic-sim` 的只读业务库配置

其中：

- 语言模型基线依赖外部可访问的 OpenAI-compatible 聊天接口。
- `cloud-device-semantic-sim` 的连接串从 `AppHost` / Aspire 资源页复制，不手写猜测值。
- 业务库配置必须保持 `isReadOnly = true`。

## 8. 权限与页面基线

- 聊天页：`/chat`
- 配置页：`/config`
- 治理页：`/access`
- 无权限页：`/forbidden`

普通 `User` 角色的 Phase 3.8 验收口径：

- 允许进入 `/chat`
- 不允许进入 `/config`
- 不允许进入 `/access`
- 访问受限页面时应跳转到 `/forbidden`

## 9. 演示前自检

开始正式演示前，先确认以下 6 项：

- `AppHost` 资源全部健康。
- 可以看到 `cloud-device-semantic-sim` 资源。
- Bootstrap 管理员可以正常登录。
- 配置页可创建并读取模型、模板、审批策略、业务库。
- 治理页可看到用户、角色、审计三个区域。
- 前端默认主链是 `/chat`，受限页面会跳转 `/forbidden`。
