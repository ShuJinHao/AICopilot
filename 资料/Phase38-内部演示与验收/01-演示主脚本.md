# Phase 3.8 演示主脚本

## 1. 演示目标

本脚本用于固定内部演示顺序，确保管理员视角、普通用户视角、配置流、聊天流、审批流、治理流、审计流都能在同一套口径下稳定复现。

推荐顺序不要改：

1. 管理员登录并查看系统基线
2. 管理员完成一轮配置流
3. 管理员完成一轮业务问答
4. 管理员完成一轮策略问答
5. 管理员完成审批通过与拒绝
6. 管理员完成治理操作并在审计中筛到结果
7. 普通用户登录后验证聊天可用、治理和配置受限

## 2. 管理员登录并查看系统基线

操作：

1. 打开登录页。
2. 使用 Bootstrap 管理员账号登录。
3. 登录后确认默认可进入 `/chat`。
4. 依次打开 `/config` 和 `/access`。

期望结果：

- 登录成功。
- 顶部导航可见聊天、配置、治理入口。
- 管理员不会被跳到 `/forbidden`。

## 3. 管理员完成一轮配置流

### 3.1 语言模型

创建或确认以下配置：

- Provider：`OpenAI`
- Name：`phase38-demo-model`
- BaseUrl：当前演示机可访问的 OpenAI-compatible 聊天接口
- ApiKey：当前演示密钥
- MaxTokens：`1024`
- Temperature：`0.2`
- IsEnabled：开启

### 3.2 通用会话模板

创建或确认以下配置：

- Name：`Phase38DemoAgent`
- Description：`Phase 3.8 内部演示通用助手`
- Model：`phase38-demo-model`
- IsEnabled：开启

### 3.3 意图路由模板

创建或确认以下配置：

- Name：`IntentRoutingAgent`
- Description：`Phase 3.8 内部演示意图路由`
- Model：`phase38-demo-model`
- IsEnabled：开启

### 3.4 审批策略

创建或确认以下配置：

- Name：`Phase38DiagnosticApproval`
- TargetType：`Plugin`
- TargetName：`DiagnosticAdvisorPlugin`
- ToolNames：`GenerateDiagnosticChecklist`
- IsEnabled：开启

期望结果：

- 审批策略列表中存在 `Phase38DiagnosticApproval`
- 策略目标准确指向 `DiagnosticAdvisorPlugin / GenerateDiagnosticChecklist`

### 3.5 业务库配置

创建或确认以下配置：

- Name：`cloud-device-semantic-sim`
- Description：`Phase 3.8 演示只读业务库`
- Provider：`PostgreSql`
- ConnectionString：从 `AppHost` / Aspire 资源页面复制 `cloud-device-semantic-sim`
- IsEnabled：开启
- IsReadOnly：开启

期望结果：

- 语义源状态显示 `Ready`
- 业务库条目标记为只读

## 4. 管理员完成一轮业务问答

进入 `/chat`，使用 `Phase38DemoAgent` 创建新会话，然后依次发送：

1. `列出 LINE-A 产线的设备`
2. `查看设备 DEV-001 的详情`
3. `查看设备 DEV-001 最新日志`
4. `查看配方 Recipe-Cut-01 的详情`
5. `查看设备 DEV-001 在 2026-04-20 到 2026-04-21 的产能`
6. `查看设备 DEV-001 最新生产记录`

期望结果：

- 每轮问答都返回结构化业务结果。
- 返回内容能稳定命中关键片段，例如 `DEV-001`、`Cutter A`、`Recipe-Cut-01`、`V2.0`、`126`、`CELL-0002`。
- 不出现写库动作或非只读行为。

## 5. 管理员完成一轮策略问答

继续在聊天页发送以下问题：

1. `系统里的权限是怎么控制的？`
2. `新用户从创建到能使用系统，需要经过哪些步骤？`
3. `用户启用、禁用、恢复这些治理动作现在怎么流转？`
4. `bootstrap 身份链在这个系统里是什么意思？`
5. `配方版本在当前语义里怎么表达？`

期望结果：

- 回答内容与当前 `AICopilot` 已对齐的规则一致。
- 回答能命中关键概念，例如 `角色`、`权限`、`BootstrapAdmin`、`审计`、`版本历史`。

## 6. 管理员完成审批通过与拒绝

操作：

1. 在聊天页发送 `please prepare a diagnostic checklist for device DEV-001`。
2. 等待出现审批卡片。
3. 选择批准。
4. 再发送 `please prepare another diagnostic checklist for device DEV-001`。
5. 等待再次出现审批卡片。
6. 选择拒绝。

期望结果：

- 第一次触发会出现审批卡片，而不是直接给出控制指令。
- 批准后，聊天中出现“已批准并生成诊断清单”的完成反馈。
- 第二次触发后，拒绝分支能返回拒绝反馈。
- 审批通过和拒绝都会留下审计记录。

## 7. 管理员完成治理操作并在审计中筛到结果

### 7.1 创建普通用户

在 `/access` 页面创建普通用户：

- UserName：`phase38-user`
- Password：`Password123!`
- Role：`User`

期望结果：

- 用户列表新增 `phase38-user`
- 角色为 `User`
- 状态为 `Enabled`

### 7.2 验证禁用与恢复

操作：

1. 禁用 `phase38-user`
2. 在审计区筛选：
   - `actionGroup = Identity`
   - `actionCode = Identity.DisableUser`
   - `targetName = phase38-user`
   - `result = Succeeded`
3. 恢复启用 `phase38-user`
4. 再筛选：
   - `actionCode = Identity.EnableUser`
   - `targetName = phase38-user`
   - `result = Succeeded`

### 7.3 验证重置密码

操作：

1. 对 `phase38-user` 执行重置密码
2. 新密码设置为 `Password123!Reset`
3. 在审计区筛选：
   - `actionCode = Identity.ResetUserPassword`
   - `targetName = phase38-user`
   - `result = Succeeded`

### 7.4 验证角色删除规则

操作：

1. 创建角色 `Phase38TempRole`
2. 将 `phase38-user` 暂时改为 `Phase38TempRole`
3. 直接删除 `Phase38TempRole`
4. 预期被拒绝后，再把 `phase38-user` 改回 `User`
5. 再次删除 `Phase38TempRole`

期望结果：

- 第一次删除被拒绝，因为角色仍有绑定用户。
- 第二次删除成功，因为角色已无绑定用户。
- 审计筛选可分别看到 `Identity.DeleteRole` 的 `Rejected` 和 `Succeeded`。

## 8. 普通用户登录后验证聊天可用、治理和配置受限

操作：

1. 退出管理员账号。
2. 使用 `phase38-user / Password123!Reset` 登录。
3. 进入 `/chat`，发送 `hello`。
4. 手动访问 `/config`。
5. 手动访问 `/access`。

期望结果：

- 普通用户可正常登录并使用聊天。
- 访问 `/config` 会跳到 `/forbidden`。
- 访问 `/access` 会跳到 `/forbidden`。
- `/forbidden` 页面应显示当前账号、当前角色，以及缺失权限提示。

## 9. 演示结束时的固定结论

如果以上 7 段脚本都能按顺序走通，则本次 Phase 3.8 内部演示通过。
