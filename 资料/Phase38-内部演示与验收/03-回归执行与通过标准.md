1. 无 Docker 本地快速门禁
2. Docker/Aspire 完整门禁

无 Docker 机器只跑第一层；完整演示与验收前，必须在 Docker 可用机器上把第二层跑绿。

- `npm run build` 通过

1. 先跑 `npm run build`
2. 再跑 `Suite=Phase43SafetyQuality&Runtime!=DockerRequired`
3. Docker 环境恢复后，再跑 `Suite=Phase38Acceptance`
4. 最后跑 `Suite=Phase43SafetyQuality`

## 6. 失败时先看什么

### 6.1 `npm run build` 失败

- Node.js 版本是否符合仓库要求
- `src\vues\AICopilot.Web\node_modules` 是否完整
- 最近是否改动了类型定义、store、服务返回 DTO

### 6.2 无 Docker 快速门禁失败

- 最近是否改动了审批策略、在岗声明、场景分类、SQL guardrail、MCP allowlist
- 错误码、审计动作码、结果截断提示是否和测试断言一致
- MCP 工具摘要是否仍只基于 `AllowedToolNames` + 已启用审批策略汇总

### 6.3 Docker 完整门禁在环境启动阶段失败

- Docker Desktop 是否已启动
- 本机能否正常拉起 `postgres`、`eventbus`、`qdrant`
- 是否存在端口占用、残留容器或 Docker daemon 不可达

说明：

- 这类失败首先视为环境阻塞，不直接视为业务代码缺陷
- 相关测试类已用 `Runtime=DockerRequired` 标识

### 6.4 聊天/审批链路失败

- `IntentRoutingAgent` 是否存在且启用
- 审批策略是否启用，`TargetType` / `TargetName` / `ToolNames` 是否匹配
- 会话是否已设置有效在岗声明
- 审批决定是否通过结构化接口提交

### 6.5 Text-to-SQL 失败

- 业务库是否仍为 `isReadOnly = true`
- 数据源是否被禁用
- SQL 是否触发只读限制、结果截断或超时

## 7. 手工验收记录方式

手工验收只记录关键片段，不做整段全文粘贴。建议记录：

- 使用账号
- 页面路径
- 示例问句或治理动作
- 命中的关键结果
- 是否通过

建议格式：

```text
[通过] /chat - 列出 LINE-A 产线设备 - 命中 DEV-001 / Cutter A / LINE-A
[通过] /chat - please prepare a diagnostic checklist for device DEV-001 - 出现审批卡片并可通过结构化审批继续执行
[通过] /access - Identity.DisableUser - 审计筛到 phase38-user / Succeeded
```
