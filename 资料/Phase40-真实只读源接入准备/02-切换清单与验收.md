# 切换清单与验收

## 切换前提

满足以下条件后，才允许进入真实源替换：

- 云端已经准备好统一只读视图或读模型。
- 五域字段契约已逐项对齐。
- `ProductionData` 已被统一抽象，不再要求 AI 侧区分不同原始表。
- 目标业务库是只读的。

## 操作顺序

1. 在配置管理中确认逻辑业务库名仍为 `DeviceSemanticReadonly`。
2. 用真实只读库连接串替换该逻辑业务库的实际连接信息。
3. 以 [appsettings.RealSource.template.json](/C:/Users/jinha/Desktop/产线系统架构升级/1/AICopilot/src/Hosts/AICopilot.HttpApi/appsettings.RealSource.template.json) 为起点，补齐五域 `SemanticMappings:*`。
4. 保证每个 target 都显式配置：
   - `DatabaseName`
   - `SourceName` 或 `FromClause`
   - `Provider`
   - `FieldMappings`
   - `AllowedProjectionFields`
   - `AllowedFilterFields`
   - `AllowedSortFields`
5. 启动 `AICopilot` 后先查看 `/api/data-analysis/semantic-source/status` 或配置页状态区，不要先做聊天联调。
6. 只有五域全部 `Ready` 后，才允许进入聊天语义验证。

## 必做验收

### 元数据验收

- 五域状态全部为 `Ready`
- `effectiveSourceName` 与预期真实源一致
- `providerMatched` 全部为 `true`
- `sourceExists` 全部为 `true`
- `missingRequiredFields` 全部为空

### 聊天回归验收

- 设备语义问答可用
- 设备日志语义问答可用
- 配方语义问答可用
- 产能语义问答可用
- 生产数据语义问答可用
- 规则问答、审批、治理链路不退化

## 失败时先看什么

- `DatabaseNotFound`：先看 `BusinessDatabase` 记录是否存在、逻辑名是否仍是 `DeviceSemanticReadonly`
- `Disabled`：先看业务库是否被停用
- `NotReadOnly`：先看业务库是否仍被保存为只读
- `ProviderMismatch`：先看 `SemanticMappings:*:Provider` 与业务库 provider 是否一致
- `SourceNotFound`：先看 `SourceName` / `FromClause` 是否能在真实库中直接执行
- `FieldMismatch`：先看五域字段契约是否补齐，尤其 `Device.status`、`Device.lineName`、`ProductionData.stationName`

## 回退条件

出现以下任一情况，直接回退到 `cloud-device-semantic-sim` 基线，不继续硬顶：

- 五域中任意一域不是 `Ready`
- `ProductionData` 无法统一成单一只读结果集
- 聊天主链开始出现为了适配真实源而新增的特判分支
- 需要修改云端原表语义才能勉强让 AICopilot 跑通
