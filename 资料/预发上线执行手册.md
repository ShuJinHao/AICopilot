# AICopilot 预发上线执行手册

更新时间：2026-04-24

## 前置条件

- 目标环境具备 Docker、PostgreSQL、Redis、RabbitMQ、Qdrant 和 AICopilot 所需模型访问配置。
- 预发环境必须提供 Redis 连接串：`ConnectionStrings__final-agent-context-redis`。
- 预发和生产必须保持 `AiGateway__FinalAgentContextStore__Provider=Redis`。
- JWT Secret、数据库连接串、模型 API Key、MCP 参数通过环境变量或部署系统注入。
- 不使用 `src/Hosts/AICopilot.AppHost/appsettings.json` 中的本地 `pg-password` 作为生产配置。

## 部署顺序

1. 部署或确认 PostgreSQL、Redis、RabbitMQ、Qdrant 可用。
2. 部署并运行 `MigrationWorkApp`，让 EF migration 和基础引导数据先完成。
3. 部署 `HttpApi`，确认启动时能连接 PostgreSQL、Redis、RabbitMQ、Qdrant。
4. 部署 `RagWorker`。
5. 部署 Web 前端资产。
6. 执行完整验收脚本并归档报告。

## Migration 验证

- 必须确认 migration `20260423064237_Phase44OnsiteAttestationColumns` 已应用。
- `sessions` 表必须存在 `onsite_confirmed_at`、`onsite_confirmation_expires_at`、`onsite_confirmed_by`。
- `onsite_confirmed_at` 和 `onsite_confirmation_expires_at` 必须是 PostgreSQL `timestamp with time zone`。
- `approval_policies` 表必须存在 `requires_onsite_attestation`，类型为 `boolean`。
- 首选验证方式是运行完整验收脚本中的 `Run Migration And Redis Verification Tests`。

## Redis 验证

- 确认 `HttpApi` 启动日志无 Redis 连接失败。
- 确认多个 HttpApi 实例共享同一 Redis 实例和同一 `final-agent-context-redis` 连接。
- 执行并发审批 smoke，确认同一 callId 只成功一次，重复请求返回 `approval_already_processed` 或等价的已处理错误。
- 若 Redis 不可用，不进入预发验收；不要把预发或生产 Provider 改回 Memory 作为规避。

## 验收命令

在 AICopilot 仓库根目录执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Run-AcceptanceClosure.ps1 -ReportPath .\资料\acceptance-closure-latest.md
```

报告中必须全部为 `PASSED`：

- Build HttpApi
- Build AppHost
- Build BackendTests
- Build Frontend
- Check Diff Whitespace
- Run Focused Unit Tests
- Check Docker
- Run Migration And Redis Verification Tests
- Run Runtime Smoke - Missing Template Error Chunk
- Run Runtime Smoke - Concurrent Approval Idempotency
- Run Runtime Smoke - Onsite Attestation Gate

## 人工抽查

- 普通聊天：能够返回文本，user/assistant 成对落库。
- 缺模板或缺模型：前端收到明确错误提示，不持续 loading。
- 审批批准：工具只执行一次，审计只写一次。
- 审批拒绝：不会执行工具，会话历史完整。
- 现场在岗过期：审批被拦截，重新声明后可继续。
- 大 SQL 结果：返回截断预览，不把完整大结果塞入模型上下文。

## 回滚步骤

- 应用回滚：将 HttpApi、MigrationWorkApp、RagWorker 和 Web 前端回滚到上一稳定版本。
- 数据库回滚：先备份数据库，再评估 migration down；没有明确批准时不删除数据库、不删除 PostgreSQL volume、不清表。
- Redis 回滚：保留 Redis 数据通常安全；如版本不兼容导致审批恢复失败，可清理 Redis 中的未完成审批上下文。
- 局部回滚：若只有 Web 显示异常，优先只回滚前端资产。

## 发布后观察

- 观察 SSE 错误率、`chat_configuration_missing` 数量、审批通过率、审批重复请求数量。
- 观察 Redis 连接错误、final-agent context 缺失、审批上下文过期数量。
- 观察 SQL Guardrail 拒绝率和大结果截断次数。
- 观察 OpenAI/模型调用失败率和 P95 延迟。
